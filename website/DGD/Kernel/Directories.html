Your MUDLib's Directory Structure

Where do I put all this stuff?

You've probably asked yourself that several times already, just getting started. When you make a new library, or a new clonable, or a new data file, where do you put it? You've got a number of choices, and many of them we won't even touch on here. Don't treat this as being the be-all and end-all on this subject, but it can get you started.

Where you put stuff is surprisingly important. The Kernel MUDLib hardcodes a lot of stuff according to paths, including a lot of information about who has access to what, and whether an object is inheritable, clonable or an LWO. So let's hit some basic guidelines.

Where do specific object types go?

You'll need to put a "/lib/" directory somewhere in the path to every library (inheritable). The Kernel MUDLib requires it. It's good practice. It's good documentation. Do it.

If you're using lightweight objects, or LWOs, you'll need to do the same with a "/data/" directory in the path. Note that this isn't required for the libraries that the LWO object inherits. Those use "/lib/" just like any other library.

There's a strong convention that clonables should have "/obj/" in their path in the same way. You don't have to do it. In fact, daemons (below) are a common, specific exception. Still, I highly recommend it in general. It's good documentation for other people looking at your MUDLib, and the Kernel MUDLib does do a few things differently depending on whether you have it.

Daemons are clonables like any others, but they're meant to be used by doing a find_object on them and calling their methods with call_other. They may be technically clonable, or may have guards in their create() routines that prevent cloning. They often wind up in a directory with "/sys/" in the name instead of "/lib/" or "/obj/". This is purely optional, mostly a form of documentation. You're just saying that you plan to use objects in that directory as a daemon.

What about Data?

This one's important too, but for different reasons. The Kernel MUDLib doesn't explicitly prevent paths from being opened with read_file(), which is the way pretty much any data file is accessed. But different paths require different permissions to access. For instance, stuff in "/usr/System" is writable only by the privileged System user by default.

You may not want to keep the user data there. While passwords are encrypted, you may need to give certain wizards read access to System but not want them to be able to read all the save files. If so, or out of sense of following conventions you should probably move it to /save, which is not only not world-readable, but has specific guards in the Kernel MUDLib for additional security. It's a great place to put private stuff that only very privileged code can access. It's meant to be.

Specific Projects

This is part of the Adding Stuff section, so what kind of stuff should you add? One simple answer is "everything", since this document tells you where everything goes. But there are also some projects you can undertake to improve your MUDLib.

For instance, a simple thing you can do is to move your user data from its current save location to "/save", as we suggest above. Mainly you'll need to modify "user.c". Go give it a try!
